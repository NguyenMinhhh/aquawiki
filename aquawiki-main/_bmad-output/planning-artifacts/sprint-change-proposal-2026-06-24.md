## Section 1 — Issue Summary

### Problem Statement

Two new product capabilities have been requested and prioritized **above** the previously planned Growth/Vision backlog:

1. **Water Change Reminder (in-app):** Notify users when a tank is due for a water change.
2. **AI Fish Identification:** Let a user photograph a fish and have the system identify the matching species from the AquaWiki database.

### Context / How Discovered

Requested directly by the product owner (NguyenMinhhh) as the **new top priority**. The owner explicitly de-scoped the remaining Growth/Vision features (bookmarks, species comparison, save/load configs, equipment management, user management, system statistics, global search, unit converter, compatible-species suggestions) — these are **no longer planned** and should not block or compete with the two new features.

### Evidence / Findings from Artifact Review

- **Tank Simulator is session-only.** Architecture (`architecture.md` line 65) and PRD (line 279) fix the MVP simulator as *not persisted to backend*. A water-change reminder requires a **persisted tank**, which does not exist today.
- **Personal Tank Management (FR38–FR42) and in-app notifications (FR45) are Growth-phase only** — no epic or story covers them in the current 5-epic plan.
- **No AI/ML integration exists.** "AI-assisted species recommendation" appears only under PRD *Vision (Future)* (line 124). Architecture has no recognition service, no external AI client, and no provider abstraction.
- **No background scheduler exists.** The fixed stack (`architecture.md` line 63) is React + Spring Boot 3.5.14 + MySQL 8 + local-filesystem images + JWT. No job scheduler is provisioned.

## Section 3 — Recommended Approach

### Selected Path: **Direct Adjustment (Hybrid)** — add two new epics; no rollback; defer (do not delete) remaining Growth/Vision FRs

**Why not Rollback:** Nothing built conflicts with the new work. No rollback is justified.

**Why not full MVP redefinition:** The 5 existing epics remain valid foundations. We extend, we don't replan from scratch.

**Hybrid element:** Epic 7 (AI) is **gated by a Technical Research spike** before its build stories are finalized, because the model/provider choice is unresolved and materially changes the architecture and cost.

### Key design decisions baked into this recommendation

1. **Minimal tank persistence, not full Personal Tank Management.** Reminders require a saved tank, so Epic 6 builds only the slim subset (create/list/edit/delete a tank + schedule fields). The broader FR40-style features stay deferred.
2. **In-app reminders via compute-on-read.** A `GET /api/reminders` (or inclusion in the tank list response) computes due/overdue status from `next_due` at request time. **No background scheduler** is added — keeps the fixed-stack architecture simple and matches the "in-app only" channel decision.
3. **Provider-agnostic AI.** A `SpeciesRecognitionService` interface isolates the chosen provider so the research outcome (external vision API vs self-hosted model vs third-party API) can be swapped without touching controllers or the frontend.

## Section 4 — Detailed Change Proposals

### 4.1 PRD Changes

**Add new Functional Requirements (active scope):**

- **FR46 (Minimal Tank Persistence):** Authenticated users can create, view, edit, and delete a saved tank record (name, dimensions/volume, optional species composition). *(Minimal subset of former FR38/FR39, promoted from Growth.)*
- **FR47 (Water Change Schedule):** Authenticated users can configure a water-change schedule for a saved tank (interval in days + last-change date); the system computes the next-due date.
- **FR48 (In-App Water Change Reminder):** The system surfaces an in-app reminder when a tank's water change is due or overdue. *(Concretizes FR45 for the water-change case; in-app channel only.)*
- **FR49 (Log Water Change):** Authenticated users can mark a water change as completed, which logs the event on the tank timeline and resets the next-due date. *(Subset of former FR42.)*
- **FR50 (AI Fish Identification):** Authenticated users can upload a fish photo and receive a ranked list of candidate species from the AquaWiki database, each with a confidence indicator, with graceful handling of no-match / low-confidence results.
- **FR51 (Identification → Species Detail):** Users can navigate from an AI identification result to the corresponding species detail page.

**Scope decision to record in PRD:**
- Promote FR46–FR49 (tank persistence + reminders) and FR50–FR51 (AI ID) into **active scope, top priority**.
- **Deprioritize / defer indefinitely** (not deleted): FR11, FR12, FR18, FR19, FR26, FR33, FR34, FR35, FR43, FR44, and the non-minimal parts of FR38–FR42, FR45. Per owner: "không cần tính nữa."

### 4.2 Architecture Changes

- **Data model:** `tanks` table (Flyway V5) with `id, user_id (FK), name, length_cm, width_cm, height_cm, volume_liters, water_change_interval_days, last_water_change_date, created_at, updated_at`. `tank_maintenance_log` table (Flyway V6) with `id, tank_id (FK), event_type, event_date, note`.
- **Reminder strategy:** Compute-on-read. `next_due = last_water_change_date + interval_days`; status derived at request time. **No `@Scheduled` job, no message queue.** (Documented as an explicit ADR for simplicity.)
- **AI service:** Add `SpeciesRecognitionService` interface (`identify(image) → List<SpeciesMatch{speciesId, confidence}>`). One provider implementation, selected after the research spike. Endpoint `POST /api/identify` (multipart), reusing AR10 image validation (MIME + size). Provider secrets via environment config (not committed).
- **Boundaries:** New backend packages `tank/` and `identification/` following the existing layered convention (`controller/service/repository/model/dto`).
